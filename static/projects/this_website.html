This Website

Summary

This is a website for purists.

The sluggish performance, common dependency hell and incredible fragmentation of modern web development prompted me to try something new. To force myself out of the current ecosystem, I set out to build a modern website without JavaScript.

I got pretty close. Aside from JavaScript bindings generated with wasm-bindgen for the necessary DOM access, this website uses only 8 lines of JavaScript. The rest is done in WebAssembly, accomplishing rudamentary client-side routing and interactivity.

The entire website -- including the Actix-based backend -- compiles to a single 9MB executable for easy deployment. You can view the source code here if you're thinking of doing something similar yourself.

The Problem

I am frequently frustrated by the ineficiency and fragmentation of modern web development. To implement client side routing for a more responsive user experience, for example, you would typically be encouraged to adopt a JavaScript framework like Vue. Let me walk you though my experience with that:

  1. Opened up the Vue v2 Guide
  2. Realized that Vue has its own ways to do:
    • Component registration (and communication)
    • Event handling
    • Forms
    • Unit testing
    • Etc.
  3. Looked for slimmer alternatives (oh hey, maybe React?).

You can imagine how my React experience went.

My beef with this situation wasn't an aversion to learning -- it was a fear that after adopting one of these technoligies, I might be stuck with it. What if it's no longer maintained in 5 years? What if I need help maintaining it, but can't find people who know my specific "stack" of frameworks anymore? What if I just want a different feature set? Commitment is scary, and web development is full of it. Making matters worse, these dependencies come with dependencies of their own and will need helper libraries and plugins to accomplish many of the tasks you require. You will likely end up with some combination of

  • vue-router
  • vuex
  • vue-custom-element
  • vue-touch
  • NPM
  • Webpack
along with any number of miscelanious dependencies you might find recommended on Stack Overflow for whatever feature you're trying to implement. Small things here and there eventually add up, and webpages have gotten big. The average webpage surpassed the original DOOM game in size back in 2016 .

There's a great writeup by HTTP Archive on why web bloat and complexity is a problem if you'd like to learn more.

The Solution

If the problem is the ecosystem as a whole pushing us towards unsustainable development practices, then solution is to get away from the ecosystem. Instead of JavaScript, this website was created using WebAssembly compiled from Rust. Tools like Bootstrap and NPM were left on the sidelines while I made my own interactive elements and relied on the Rust-native dependency manager and build tool, Cargo.

WebAssembly, however, lacks many of the privilages of JavaScript (like DOM access). To get around this, a total of 8 lines of JavaScript were used to load the WebAssembly, and some more auto-generated as glue code by wasm-bindgen (mentioned earlier). These 8 lines load the glue and pass events to my WebAssembly. 

    
// Import the init function from the 
// generated JS bindings and call it.
import init, * as busy from '/api/bindings';
async function run() {
  // Load the wasm file
  await init('/api/wasm');
}
run();

// Store wasm functions in window to make 
// them accessible outside this script.
window.busy = busy;

// Add popstate event handler.
window.onpopstate = function(event) {
  if (event.state) {
    window.busy.route(event.state);
  }
};

With this out of the way, I was able to start working on the fun stuff. Routing boiled down to a very simple function run on every request. 

    
/// Get current URL and load the resulting page.
#[wasm_bindgen]
pub fn route(rt: &str) {
    match rt {
        "/" => spawn_local(welcome()),
        "/welcome" => spawn_local(welcome()),
        "/resume" => spawn_local(resume()),
        "/contact" => spawn_local(contact()),
        "/projects/this_website" => 
            spawn_local(this_website()),
        _ => spawn_local(error_404()),
    }
}

Meanwhile, browser history is manipulated from within each page's associated function as shown below for the welcome page. 

    
if history
    .state().expect("Could not get history state") != "/welcome" {
    history
        .push_state_with_url(
            &JsValue::from_str("/welcome"), 
            "Welcome!", 
            Some("/welcome"))
        .expect("Could not push state (with URL) to history");
}

The actual content of pages (when not loaded on first contact) is retrieved as-needed by HTTP request. 

    
let mut req = RequestInit::new();
req.method("GET");
let request =
    Request::new_with_str_and_init("/api/welcome", &req)
        .expect("Request could not be created");
request
    .headers()
    .set("Accept", "text/html")
    .expect("Headers could not be set");

let response = JsFuture::from(window.fetch_with_request(&request))
    .await
    .expect("Could not unwrap response");

// `response` is a `Response` object.
assert!(response.is_instance_of::<Response>());
let resp: Response = response.dyn_into().unwrap();

// Convert this other `Promise` into a rust `Future`.
let page = JsFuture::from(resp.text().unwrap())
    .await
    .unwrap()
    .as_string()
    .unwrap();

// Show the new content.
document
    .get_element_by_id("page")
    .unwrap()
    .set_inner_html(&page);

The frontend is served by an actix-web backend, with the entire frontend compiled seperately and embedded in the final executable using the include_bytes!() macro. while this method is less than ideal for compile times, it makes single-file deployment a possibility.

Closing Remarks

This has been a fun project, and I plan on using this website to host my projects for a long time. That said, I would never use a setup like this for a serious production website. As cool as this is, it's just too far from standard.
